# 設計理念

ここでは，Yuライブラリの設計理念を述べる．
これはすべての詳細規則に優先する上位設計原則を定義する．
個別の設計判断において規則間の衝突が生じた場合，本理念に基づいて解釈されるものとする．


## 目的

Yuライブラリは，C++の汎用ライブラリを，単なる機能の集合ではなく意味論的な抽象の体系として構築することを目的とする．
Yuライブラリは特定の領域を対象とするフレームワークではなく，言語機能を拡張するための小さく独立した抽象を提供するライブラリ群である．

## 基本原則

### 意味論中心の設計

各サブライブラリは機能の分類ではなく意味論と責務によって分割される．

サブライブラリは次を満たす：

* 独立した概念を定義する
* 単独で理解可能である
* 他ライブラリに対して意味論的基盤または拡張を提供する

単なる分類や整理のためにサブライブラリを分割してはならない．

### 併用可能性

各サブライブラリは相互に組み合わせられることを前提に設計される．
依存関係はエンティティの共有のためではなく，意味論の強化のためにのみ導入される．

### 拡張可能性

Yuライブラリは利用者による拡張可能性を重視する
そのため，コンセプトやCPO，ADLによる拡張を中心とした設計を採用する．
拡張は既存コードの変更を要求してはならない．

### ゼロオーバーヘッド原則

Yuライブラリが提供するすべての機能は，可能な限り追加コストを導入しないことを目標とする．
つまり，既存のコードをYuライブラリを用いたものに置き換えたとしても，可能な限りオーバーヘッドを増やしてはならない．
ただし，可読性および意味論的明確性を犠牲にしてはならない．

### ヘッダオンリー設計

Yuライブラリはヘッダオンリーの形で提供される．

これは，
* 導入コストの最小化
* 部分的取得の容易化
* ABI問題の回避
を目的とする．

ビルドシステムへの依存は設計上の前提としない．

## ライブラリ構造

### サブライブラリ

Yuライブラリは単一の巨大ライブラリではなく，複数のサブライブラリの集まりとして設計される．

各サブライブラリは，独立した公開名前空間を持ち，明確な責務の境界を持つ．
また，許可される依存関係の例外を除き，個別に取得可能でなくてはならない．

### 基盤ライブラリ

一部のサブライブラリは，他ライブラリの基盤となる機能を提供する．
これらは広く依存されることを前提とするが，依存先へ逆方向の依存を持ってはならない．

## 公開境界

Yuライブラリは，公開APIと非公開実装を明確に分離する．
利用者はリファレンスで公開されたエンティティのみに依存でき，それ以外の非公開のエンティティへの依存は互換性を保障しない．

## 更新方針

### 安定性と互換性

Yuライブラリは，原則として後方互換性を維持する．

既存APIを意味論的に変更することは避けられ，新機能追加は既存コードに影響を与えてはならない．
破壊的変更は原則として許可されない．

### 更新

Yuライブラリは継続的に更新される．
